# Future Roadmap

## 1. 将来性の評価

このプロジェクトは、現時点でまだコンテンツ量こそ限定的ですが、将来性は高いです。理由は単純で、ゲームの核となる体験がすでに明確であり、しかもその核が拡張しやすい構造で実装されているからです。

特に将来性が高い要素は以下です。

- プレイヤー間リンクという独自性の高いコアメカニクス
- ScriptableObject によるデータ駆動構成
- マルチプレイ導線の実装済み基盤
- 責務分離されたスクリプト構成
- ランタイム modifier ベースのステータス設計

これらは、あとから追加されるコンテンツを受け止めるための土台として優秀です。

## 2. 拡張性が高い理由

### 2.1 リンク中心ゲームデザイン

本作の独自性は「プレイヤーの位置関係が攻撃そのものになる」点です。これは今後、以下のようにいくらでも派生させられます。

- 2 人リンクと 3 人リンクで効果が変わる
- リンクの色や属性を持たせる
- リンクを曲線化、分岐化する
- 特定距離帯でクリティカル化する
- リンク上を弾やエフェクトが流れる

つまり、単に敵や武器を増やすよりも、リンク周りを伸ばすだけでゲーム性を深くできます。

### 2.2 データ追加で中身を増やしやすい

現状でも、ステージやアップグレードは asset を増やす方向で拡張できます。これは少人数開発で特に重要です。

### 2.3 オンライン前提の土台がある

後からマルチプレイを載せるより、最初から Relay / Lobby を見据えている方が圧倒的に強いです。このプロジェクトはすでにその段階を越えています。

## 3. 短期的に追加すると効果が大きいもの

### 3.1 アップグレードの種類追加

優先度が高い理由:

- 現在のゲームループに自然に乗る
- ScriptableObject 基盤をそのまま活かせる
- プレイバリエーションがすぐ増える

候補:

- Link 距離増加
- Link 幅増加
- Link 攻撃間隔短縮
- Core 自動回復
- 敵撃破時の一時バフ
- プレイヤー移動速度増加

### 3.2 敵タイプ追加

候補:

- 高速接近型
- 自爆型
- シールド持ち
- リンクを遮断する妨害型
- Core 優先特攻型

特に「リンクに対してどう振る舞うか」を差別化できる敵は、本作らしさを強くできます。

### 3.3 HUD 強化

候補:

- Core HP 表示
- 現在ウェーブ
- 接続人数
- 現在アクティブなアップグレード一覧
- プレイヤー別選択状況

これは実装難度の割に体験向上が大きいです。

## 4. 中期的な拡張案

### 4.1 ビルドシナジー

アップグレード同士に組み合わせ効果を持たせると、プレイヤー間の相談や投票がより意味を持ちます。

例:

- Link 幅強化 + 攻撃力強化で貫通判定追加
- Core HP 増強 + 回復系で再生バリア化

### 4.2 プレイヤーロールの分化

例:

- Support: Link 範囲拡張に強い
- Tank: Core 防衛補助が強い
- Assault: Link 攻撃性能に特化

現状の PlayerConfig をクラス別に増やしていく方向でも実現しやすいです。

### 4.3 ステージギミック

例:

- リンクを反射する壁
- 特定地点でリンク強化
- 敵スポーン抑制装置
- 移動制限フィールド

これにより、単なる敵数増加ではない戦術差を作れます。

## 5. 長期的な将来像

### 5.1 コンテンツ量の拡張

将来的には以下まで伸ばせます。

- 複数ステージ
- 複数ゲームモード
- ボス戦
- Endless モード
- ランダムイベント
- プレイヤー成長のメタ進行

### 5.2 運用面の拡張

オンライン前提なら、以下も視野に入ります。

- ロビー設定拡張
- フレンド招待
- ルームコード共有改善
- 切断復帰
- セッション再開

## 6. 技術面で今後整えると強い点

### 6.1 プレイヤー登録管理の明確化

現在は `Movement` 探索でプレイヤー収集していますが、将来的には PlayerRegistry のような明示的登録型へ寄せると安全です。

### 6.2 ステータス層の共通化

LinkRuntimeStats と同じ思想で、移動速度、耐久、範囲、クールダウンなどもランタイムステータス化すると、アップグレード追加がさらに楽になります。

### 6.3 エラー処理と復旧

マルチプレイの実運用を考えるなら、以下が重要です。

- Join 失敗時の UI 表示
- Relay 切断時の再試行
- ロビー消滅時の遷移先制御
- タイムアウト時の戻り先統一

### 6.4 シーン依存の削減

現在でも良い分離はありますが、Battle シーン上の参照が増えてくると導入ミスが起きやすくなります。初期化ハブや installer 的な構成に寄せる余地があります。

## 7. 優先度順の推奨ロードマップ

### Phase 1: すぐ効く改善

- アップグレード追加
- 敵追加
- HUD 強化
- 効果音 / 演出の強化

### Phase 2: プレイ体験の差別化

- Link 属性
- プレイヤーロール
- ステージギミック
- ボス敵

### Phase 3: 継続運用向け改善

- 再接続
- ロビー設定拡張
- マッチメイキング改善
- メタ進行

## 8. README に追記すべきだと考えた観点

今回の整理に加えて、今後 README または docs に継続的に残すべきなのは以下です。

- Battle シーンに必要な参照一覧
- 新しい敵 prefab を追加する手順
- 新しい upgrade asset を追加する手順
- ネットワークデバッグ時の確認項目
- Build 前チェックリスト

## 9. 総括

LINK-SLICE は、ゲームの核が成立しているうえで、伸ばす余地がかなり大きいプロジェクトです。特にリンクという固有メカニクスは、コンテンツ追加だけでなく、ゲーム全体の方向性そのものを広げられる強い基盤です。

今後は「アップグレード数を増やす」「敵の対リンク挙動を増やす」「HUD とネットワーク運用を強化する」の 3 本柱で進めると、短期の手応えと長期の拡張性を両立しやすいです。
