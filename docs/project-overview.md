# Project Overview

## 1. ゲームの概要

LINK-SLICE は、プレイヤー同士の位置関係そのものを攻撃リソースに変換する、協力型のウェーブ防衛アクションです。

このゲームの中心的な特徴は、各プレイヤーが独立して移動するだけではなく、一定距離内にいる他プレイヤーと自動的にリンクされ、そのリンクが敵への攻撃判定を持つことです。つまり、一般的なシューティングやアクションのように個人が弾を撃つ設計ではなく、プレイヤー同士の隊形、距離感、分散と集合の判断が戦術そのものになります。

現状の実装を見る限り、本作は以下の体験を軸に設計されています。

- 複数人で位置取りを行う協力プレイ
- プレイヤー間リンクを利用した範囲攻撃
- ウェーブ進行型の耐久戦
- ウェーブ間のアップグレード選択によるビルド成長
- Lobby / Relay を経由したオンライン接続

## 2. 現在のプレイサイクル

コード上で確認できる現状の基本ループは以下です。

1. Lobby シーンでホストがロビーを作成する
2. 参加者がロビーへ参加する
3. ホストがゲーム開始を実行する
4. Relay 接続と Netcode の接続人数待機を経て Load シーンへ入る
5. Load シーンから Battle シーンへ遷移する
6. Battle シーンで敵ウェーブが進行する
7. 各ウェーブ突破後にアップグレード UI を表示する
8. 選択結果を反映して次ウェーブへ進む

この流れは、プレイヤーの移動、リンク生成、敵生成、アップグレード、ネットワーク遷移がそれぞれ独立コンポーネントで構成されつつ、全体としては一つのループに統合されている点が特徴です。

## 3. 現在実装されているコアゲームプレイ

### 3.1 プレイヤー移動

`Assets/Scripts/Player/Movement.cs` では、プレイヤーがマウスカーソルに向かって移動します。

- `NetworkBehaviour` を継承
- `IsOwner` 判定によって自分のプレイヤーのみ操作可能
- カメラ移動範囲内に位置を clamp

この設計により、入力処理はクライアントごとの所有オブジェクトに閉じ、リンクや敵 AI のような共有ロジックとは分離されています。

### 3.2 プレイヤー間リンク

`Assets/Scripts/Player/Link/LinkController.cs` はシーン中の `Movement` を持つオブジェクトを収集し、プレイヤー同士の距離を毎フレーム判定してリンクの生成・維持・破断を行います。

現状の特徴:

- 一定距離以内に入るとリンク生成
- 順序に依存しないペアキーで 1 組につき 1 本だけ管理
- 距離外になった場合は即削除ではなく破断演出後に破棄
- 各リンクに共有ランタイムステータスを注入

この構造により、プレイヤー人数が増えても、リンク生成責務が単一マネージャーに集約されているため、表現の一貫性を保ちやすくなっています。

### 3.3 リンク攻撃

`Assets/Scripts/Player/Link/Link.cs` と `Assets/Scripts/Player/Attack.cs` により、リンク自体が敵へダメージを与えます。

- `LineRenderer` で視覚化
- 線分の中心、角度、距離から帯状の当たり判定を算出
- `Physics2D.BoxCastAll` により対象を取得
- `IDamageable` 実装に対してダメージを適用
- ダメージ量は `LinkRuntimeStats` の共有値から取得

これにより、「リンクの見た目」と「リンクの攻撃性能」が分離されています。見た目の調整は `PlayerConfig`、火力は `LinkRuntimeStats` で扱えるため、将来的な調整がしやすいです。

### 3.4 敵ウェーブ

`Assets/Scripts/Enemy/EnemySpawnController.cs` では、設定済みの `EnemySpawnConfig` を順番に処理して敵をスポーンします。

確認できる実装内容:

- 複数の `EnemySpawnConfig` を順に消化
- 各設定内に複数ウェーブを定義可能
- 各ウェーブ内に複数 composition を定義可能
- spawn pattern は `Random` と `Grouped` をサポート
- 敵が全滅するまで待機し、その後に次ウェーブへ進行
- 設定ブロックの合間にアップグレード UI を表示

現状のデータアセットでは、`Assets/Datas/Stage/Stage1.asset` と `Assets/Datas/Stage/Stage2.asset` が存在し、近接型と遠距離型の敵を組み合わせた段階的な進行が組まれています。

### 3.5 アップグレード選択

`Assets/Scripts/Upgrade/UpgradeManager.cs` は、アップグレード候補の提示、投票受付、結果反映を担います。

現状の仕様:

- 提示数は 3 つ
- `UpgradeDatabase` から提示可能な定義のみを抽出
- 各クライアントが 1 つ選択
- 選択人数を UI に反映
- 全員選択後、選ばれた候補群からランダムに 1 つを最終適用

この仕様は「多数決」ではなく「全員が候補を提示し、その中から最終的に 1 つ選ばれる」性質を持っています。ここは将来的に設計変更しやすい部分でもあります。

## 4. シーン構成

### 4.1 Lobby

ネットワークセッションの入口です。

- ロビー作成
- ロビー一覧更新
- ロビー参加
- ゲーム開始

Lobby の責務は戦闘ではなく、セッション組成です。これにより Battle 側の責務を軽く保てています。

### 4.2 Load

単なる遷移用ではなく、Netcode のシーン同期を崩さず本編へつなぐ中継シーンです。

`Assets/Scripts/Common/Loader/SceneLoaderManager.cs` と `Assets/Scripts/Common/GlobalCommon.cs` の実装から、Battle へ直接飛ぶよりも Load を経由することが前提になっています。

### 4.3 Battle

プレイ本体です。

- プレイヤー移動
- リンク生成
- 敵 AI
- ウェーブ進行
- アップグレード選択
- Core の体力更新

## 5. ネットワーク構成

現状は Unity Services を利用したオンラインマルチプレイ構成です。

- Authentication
- Lobby
- Relay
- Netcode for GameObjects
- Unity Transport

`Assets/Scripts/Relay-Lobby/LobbyManager.cs` がロビー制御の入口、`Assets/Scripts/Relay-Lobby/RelayTest.cs` が実際の Relay 設定と Host / Client 起動を担当します。

Battle シーンへ入るまでは PlayerObject の自動生成を抑止し、Battle 到達後に手動生成する設計になっているため、シーン遷移時の不整合を回避しやすい構成です。

## 6. 採用技術の意味

### 6.1 Netcode for GameObjects

ゲームの共有状態とマルチプレイ同期の土台です。

- `NetworkBehaviour`
- `NetworkVariable`
- `ClientRpc`
- PlayerObject 管理

### 6.2 UI Toolkit

アップグレード UI は UXML / USS ベースで構成されています。

- `Assets/UI Toolkit/Main_Upgrade.uxml`
- `Assets/UI Toolkit/Temp_UpgradeElement.uxml`
- `Assets/UI Toolkit/Style/UpgradeElement_Style.uss`

これにより、表示ロジックと見た目を比較的分離しやすい状態です。

### 6.3 ScriptableObject

ゲーム設定とアップグレード定義をコードから切り離すために使われています。

- プレイヤー基本設定
- 敵ウェーブ設定
- アップグレード定義
- アップグレード一覧

この設計は、バランス調整と機能追加の両方に有利です。

## 7. このゲームの強み

現状の実装だけでも、本作には以下の強みがあります。

- プレイヤー間リンクという独自の攻撃概念が明確
- 個人戦術ではなく協力配置がゲーム性に直結している
- データ駆動でステージ・強化要素を増やせる
- オンライン導線がすでに存在し、単なるプロトタイプ以上の構造を持つ
- 主要責務が分離されており、機能追加の足場がある

## 8. 現段階での位置付け

現状は「遊べる基礎ループが成立しているネットワーク対応アクションの土台」です。完成品というよりは、ゲームの核となる体験が明確に成立し始めている段階です。

特に優れているのは、リンク、アップグレード、敵スポーン、ロビー遷移が、それぞれ将来の追加に耐える単位で実装されている点です。このため、機能を積み上げていく余地が大きく、将来性の高い土台になっています。
